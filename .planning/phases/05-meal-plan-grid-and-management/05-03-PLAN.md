<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser slot action -> tRPC mutation | Destructive and AI-triggering actions originate from untrusted client controls |
| tRPC mutation -> LLM + database writes | Regeneration and deletion change persisted user data and must stay scoped to one owned slot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | T | `meal.delete` | mitigate | Validate the meal belongs to the caller's plan before delete and re-query the plan so only the targeted slot becomes empty |
| T-05-08 | E | `meal.regenerate` | mitigate | Accept only `mealPlanId`, `dayOfWeek`, and `mealType`, resolve the plan under RLS, and replace one matching slot instead of allowing arbitrary meal writes |
| T-05-09 | D | slot-local mutations | mitigate | Keep pending/error state local to one slot so a failed mutation cannot lock the whole page or clear sibling slots |
</threat_model>
