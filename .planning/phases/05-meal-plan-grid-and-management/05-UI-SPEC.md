# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual Tailwind + CSS custom properties) |
| Preset | not applicable |
| Component library | none — bespoke React components built on Tailwind utilities |
| Icon library | none required for Phase 5; prefer text labels over icon-only actions |
| Font | Newsreader (display/headlines) + Manrope (UI, labels, body) |

Source: `src/styles/globals.css`, `tailwind.config.ts`, Phase 4 UI contract, repo constraint to preserve existing editorial language.

Phase 5 must inherit the existing editorial shell:
- Keep the warm paper background, soft glass cards, sage action color, and generous spacing.
- Do not introduce shadcn, Radix primitives, or a denser admin/dashboard visual language for this phase.
- The meal-management UI should feel like an extension of the current plan page, not a new subsystem.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, confirmation text spacing |
| sm | 8px | Compact action spacing, chip padding |
| md | 16px | Default internal spacing, button gaps |
| lg | 24px | Card padding, section gaps |
| xl | 32px | Major layout gaps between plan sections |
| 2xl | 48px | Flyout section spacing, page rhythm |
| 3xl | 64px | Top-level page separation |

Exceptions:
- Minimum touch target: `44px` height for all interactive actions, including `View details`, `Regenerate meal`, `Delete meal`, `Keep meal`, and `Confirm delete`
- Card border radius: `1.5rem` to `1.75rem`; stay aligned with existing card shell and AppFrame curves
- Flyout width may use a non-scale cap for usability: `min(32rem, 100vw)` on smaller desktops and `max-w-[36rem]` on larger screens

## Typography

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Body | 14px | 400 | 1.75 | Manrope | Description text, rationale text, empty-slot body, supporting copy |
| Label | 14px | 600 | 1.25 | Manrope | Button labels, section labels, meal metadata, inline confirmation prompt |
| Heading | 20px | 600 | 1.2 | Newsreader | Flyout section headings, day headings on mobile |
| Display | 30px to 36px | 400 | 1.0 | Newsreader | Flyout meal title, page-level inherited plan heading |

Rules:
- Keep weights to exactly two values: `400` and `600`.
- Meal card title remains display-led but should step down from the page hero. Use `font-display text-xl leading-snug text-[var(--color-sage-deep)]`.
- Destructive confirmation text uses body size, not oversized warning treatment.
- Rationale copy should remain readable, not code-like: `text-sm leading-7 text-[var(--color-muted)]`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#faf9f8` / existing surface gradient | App background, flyout backdrop base, overall canvas |
| Secondary (30%) | `rgba(255,255,255,0.52)` and `rgba(255,255,255,0.72)` | Grid shell, meal cards, flyout panel, nav pills |
| Accent (10%) | `#4A6741` (`--color-sage`) | Reserved interactive emphasis only |
| Destructive | `#803b26` | Delete confirmation text, destructive button state, inline destructive surface |

Accent reserved for:
- `Create new plan` CTA
- `Regenerate meal` primary action on a filled or empty slot
- Active/focused flyout trigger styling
- Slot-local loading affordance for regeneration
- Keyboard focus outlines and selected-action emphasis

Accent must not be used for:
- Body copy blocks
- Card backgrounds
- General borders
- All actions indiscriminately

Destructive usage:
- Inline delete confirmation surface
- `Confirm delete` button
- Destructive helper text

Neutral text colors remain unchanged from Phase 4:
- `--color-ink` for primary text
- `--color-muted` for supporting text
- `--color-sage-deep` for headings and high-emphasis editorial labels

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Create new plan` |
| Filled slot secondary CTA | `View details` |
| Filled slot primary management action | `Regenerate meal` |
| Empty slot heading | `Open slot` |
| Empty slot body | `This meal was removed. Regenerate this slot to replace it.` |
| Empty slot CTA | `Regenerate meal` |
| Flyout section label | `Meal details` |
| Rationale section heading | `Why this fits` |
| Error state | `We couldn't update this meal. Try again in a moment.` |
| Delete confirmation | `Delete meal:` `Remove this meal from the plan? This leaves the slot open until you regenerate it.` |

Additional copy rules:
- Avoid edit language such as `Edit title`, `Rename`, or `Save changes` in Phase 5.
- Regeneration copy should emphasize replacement, not retrying the page.
- Empty slots must read as intentional states, not missing data or failed rendering.
- Flyout copy should be calm and explanatory, not technical.

## Component Inventory

### `PlanPage` updates
- `/plan/:id` becomes a persisted management surface, not only a generation result screen.
- When the user revisits an existing plan, hydrate the grid from saved DB state before showing any management affordance.
- Add a visible `Create new plan` action near the plan heading. It is secondary to the grid but always available.
- Keep page-level loading and error banners distinct from slot-local actions.

### `MealPlanGrid`
- Preserve the validated Phase 4 layout:
  - Mobile: stacked day sections with visible day headings
  - Desktop: day columns and meal-type rows
- Empty slots remain rendered in-grid after deletion.
- Each cell must render one of four states only:
  - `filled`
  - `empty`
  - `regenerating`
  - `error`
- Do not collapse or remove a slot after deletion.

### `MealCard`
- Keep existing editorial shell and elevate it into a management card.
- Recommended structure:
  - Eyebrow: meal type
  - Title: meal title
  - Short description: 2 to 3 lines visible in-grid
  - Actions row: `View details`, `Regenerate meal`, `Delete meal`
- Action row behavior:
  - Show as quiet text/button controls, not icon-only chrome
  - On mobile, stack or wrap cleanly without compressing below `44px` height
  - `Regenerate meal` gets accent treatment
  - `Delete meal` starts quiet and only turns destructive inside confirmation mode

### `EmptyMealSlot`
- New dedicated component or explicit render path within `MealCard`.
- Must visually share the same footprint as a filled card so the grid stays stable.
- Content:
  - Eyebrow: meal type
  - Heading: `Open slot`
  - Body copy from copywriting contract
  - Primary action: `Regenerate meal`
- Empty slot styling should be slightly lighter than a filled card but still intentional and complete.

### `MealRegeneratingCard`
- Slot-local loading state used for both filled and empty slots.
- Must replace the slot in place without blocking the rest of the page.
- Visual treatment:
  - Keep the card shell visible
  - Dim existing content or placeholder content to about 60% opacity
  - Add a subtle pulse on an accent-tinted loading line or status chip
  - Status text: `Regenerating meal...`
- Do not use a page overlay, spinner-only blank state, or full-page banner for this action.

### `MealDeleteConfirmation`
- Inline confirmation region that appears inside the card where actions normally sit.
- Required actions:
  - `Keep meal`
  - `Confirm delete`
- Confirmation should not open a modal or the flyout.
- On `Keep meal`, restore the normal actions row without side effects.

### `MealDetailFlyout`
- Right-side slide-over panel anchored to the viewport edge.
- Opens over the page while leaving the grid visible behind it.
- Required content:
  - Meal title
  - Meal type + day/slot context
  - Short description
  - Rationale
  - Slot actions: `Regenerate meal`, `Delete meal`, close affordance
- Structural requirement:
  - Design the content blocks so Phase 6 can append enriched recipe sections without rebuilding the shell.
- Recommended sections:
  - Header
  - Summary
  - `Why this fits`
  - Management actions

## Interaction States

### Grid-level page states

| State | What the user sees |
|-------|--------------------|
| Initial persisted load | Existing plan heading and grid shell; cards appear as loaded content, not generation skeletons |
| No meals in a saved plan | Full grid still renders with intentional empty slots and visible `Regenerate meal` actions |
| Page load failure | Editorial error banner above grid with retry path; no destructive wording |

### Filled meal card states

| State | Behavior |
|-------|----------|
| Resting | Card shows title, description, and three actions: `View details`, `Regenerate meal`, `Delete meal` |
| Hover / focus | Slight contrast lift or shadow lift only; do not imply inline editing |
| Delete armed | Actions collapse into inline confirmation copy plus `Keep meal` and `Confirm delete` |
| Regenerating | Card stays in place, actions disabled, local loading copy visible |
| Mutation error | Card returns to prior content with inline error text and retry affordance for regeneration |

### Empty slot states

| State | Behavior |
|-------|----------|
| Resting | Visible empty-state card with explanatory copy and `Regenerate meal` |
| Hover / focus | Slight emphasis to indicate the slot remains actionable |
| Regenerating | Empty slot transforms into local loading state in the same footprint |
| Mutation error | Empty slot remains visible with inline error copy and retry action |

### Flyout states

| State | Behavior |
|-------|----------|
| Open | Panel slides from the right; body scroll remains anchored to the plan page |
| Close | Close via explicit button, `Escape`, backdrop click, or route/state reset |
| Regenerating from flyout | Flyout stays open and reflects local loading; matching grid slot updates in parallel |
| Delete from flyout | Requires inline confirmation inside flyout action area; if confirmed, flyout stays open long enough to show the slot is now empty or closes immediately after the mutation if that proves simpler, but the grid must remain visibly updated |

Accessibility requirements:
- Flyout traps focus while open and returns focus to the originating card action on close.
- Keyboard users must be able to trigger `View details`, `Regenerate meal`, and delete confirmation without pointer-only affordances.
- Destructive confirmation must be announced with clear text, not only color changes.

## Layout Guidance

### `/plan/:id` page composition

```text
AppFrame
└── main
    └── space-y-8
        ├── plan header card
        │   ├── eyebrow
        │   ├── page heading
        │   ├── supporting copy
        │   └── Create new plan action
        ├── optional page-level banner
        └── MealPlanGrid
            ├── mobile stacked day sections
            └── desktop day-column grid
```

Header guidance:
- Reuse the existing plan-page shell styling from Phase 4.
- Replace generation-first language with management-first language when a persisted plan is loaded.
- The weekly meal grid is the primary visual anchor on `/plan/:id`; `Create new plan` stays visible but intentionally secondary to the current plan.
- `Create new plan` should sit in the header card, not buried beneath the grid.

### Flyout layout

```text
viewport
├── dimmed page content with grid still visible
└── right-side panel
    ├── sticky header with title + close
    ├── scrollable content body
    └── sticky actions footer or persistent actions block
```

Flyout sizing:
- Mobile: full-width overlay from the right
- Tablet and desktop: partial-width panel, approximately `28rem` to `36rem`
- Backdrop should be translucent, not opaque black; preserve awareness of the grid behind it

## Visual Rules

- Preserve the existing glass editorial language:
  - translucent white surfaces
  - soft blur
  - sage-led emphasis
  - serif display moments for titles
- Management actions must feel lightweight and local.
- Do not introduce:
  - dense table chrome
  - checkbox-heavy admin interactions
  - filled red danger cards as the default delete treatment
  - inline editable text fields on cards
  - center modal detail views

Motion:
- Flyout enter: `200ms` to `240ms` ease-out slide/fade
- Flyout exit: `160ms` to `200ms` ease-in
- Slot regeneration: subtle opacity/status transition only; avoid dramatic skeleton resets
- Delete confirmation: content swap, not a jarring layout jump
