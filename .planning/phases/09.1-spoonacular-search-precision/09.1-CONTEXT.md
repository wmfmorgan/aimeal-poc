# Phase 9.1: Spoonacular Search Precision - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning
**Source:** PRD Express Path (`.claude/skills/spike-findings-aimeal-poc/references/spoonacular-search-precision.md`)
**Origin:** Synthesizes validated spike findings 006, 007, 008 into one coupled implementation.

<domain>
## Phase Boundary

Implement the spoonacular-search-precision blueprint:

1. Map UI-collected household data → Spoonacular `complexSearch` parameters (cuisine, intolerances, diet, excludeIngredients, equipment, maxReadyTime).
2. Extend the LLM draft schema with a `search_hints` object (cuisine, type, include/exclude ingredients, max_ready_time_min, main_ingredient) and persist it on each meal row.
3. Collapse the existing two-call enrichment flow (`complexSearch` → `recipes/{id}/information`) into a single `complexSearch` call with `addRecipeInformation=true&addRecipeNutrition=true`.

**In scope:**
- Server-side enum constants & mappers (cuisine/type/diet/intolerance/equipment).
- Household restriction aggregator (members → flat restriction object).
- LLM prompt updates in `generate-draft` + `trpc` regenerate paths.
- Server-side hint normalizer (drop-on-invalid).
- New `meals.search_hints jsonb` column + Supabase migration.
- Replacement of `findBestSpoonacularMatch` + variant ladder with single consolidated `searchAndEnrich`.
- Removal of the second usage event tagged `recipes/{id}/information`.
- Three soft KPI counters in dev diagnostics (first-call hit rate, enum drop rate, empty instructions rate).

**Out of scope:**
- Replacing `diet_type` UI free-text field with a curated dropdown (deferred — backend fuzzy-mapping is the stopgap).
- Switching to `response_format=json_schema` strict mode (deferred to a future hardening spike).
- Bumping `number` above 1 in `complexSearch` (no telemetry justification yet).
- Removing the post-fetch `extractBlockedTerm` filter (kept as defense-in-depth).

</domain>

<decisions>
## Implementation Decisions

### LOCKED — Source of Truth Boundaries
- **LLM (`grok-4-1-fast-non-reasoning`) is the source of truth for recipe-specific search hints.** Per-recipe cuisine, type, ingredients, ready-time live in the LLM output.
- **Household table is the source of truth for cross-recipe restrictions.** `intolerances`, `excludeIngredients`, `diet`, `equipment` come from aggregated household members.

### LOCKED — Enum Tables (Server-Side)
Hardcode these sets in `_shared/spoonacular-mappings.ts` (or analog):
- `CUISINE_ENUM` — 26 values (african through vietnamese — full list in PRD).
- `TYPE_ENUM` — 14 values (main course, side dish, dessert, … drink).
- `DIET_ENUM` — 11 values (gluten free, ketogenic, vegetarian, …, whole30).
- `INTOLERANCE_ENUM` — 12 values (dairy, egg, gluten, …, wheat).
- `EQUIPMENT_ENUM` — subset of ~12 values matching what UI dropdown actually offers.
- `MEAL_TYPE_TO_SPOONACULAR` map: `breakfast→breakfast`, `lunch→main course`, `dinner→main course`.

### LOCKED — Hard vs Soft Filter Boundary
- **HARD filters (never relaxed in retries):** `diet`, `intolerances`, `excludeIngredients`. A "no match" outcome is correct when household constraints can't be met.
- **SOFT filters (dropped on retry):** `equipment`, `maxReadyTime`. One retry path; if hard filters still return zero, return null.
- **Always present:** `query=meal.title`, `number=1`, `instructionsRequired=true`, `sort=popularity`, `addRecipeInformation=true`, `addRecipeNutrition=true`.

### LOCKED — Household Aggregation Rules
- `intolerances`: union across members; allergies matching enum → `intolerances`, non-enum allergies → `excludeIngredients`.
- `excludeIngredients`: union of all member `avoidances` + non-enum allergies (lowercased, trimmed).
- `diet`: single enum value when ALL members agree on same diet; otherwise null. Never pass partial multi-member diet.
- `equipment`: lowercase appliance names that match `EQUIPMENT_ENUM`; non-matches dropped.
- `maxReadyTime`: derived from skill level — `beginner=30`, `intermediate=60`, `advanced=null`.

### LOCKED — LLM Schema Extension
Per-meal NDJSON line gains a `search_hints` object:
```json
"search_hints": {
  "main_ingredient": "...",
  "include_ingredients": ["...","..."],
  "cuisine": "<enum value or empty string>",
  "type": "<enum value>",
  "max_ready_time_min": 30,
  "exclude_ingredients": ["..."]
}
```
Constraints embedded in prompt: include_ingredients 1–4 items, max_ready_time_min integer 5..120, exclude_ingredients are recipe-specific (in addition to per-household avoidances).

Updates BOTH `buildSystemPrompt` (`generate-draft/index.ts`) AND `buildSingleSlotSystemPrompt` (`trpc/index.ts`).

### LOCKED — Hint Normalization (Drop-on-Invalid)
- `cuisine`/`type`: trimmed lowercase, dropped to null if not in enum. Never coerce/fuzzy-match.
- `includeIngredients`: max 4 strings, lowercased, trimmed, empty filtered.
- `excludeIngredients`: max 8 strings, lowercased, trimmed, empty filtered.
- `maxReadyTime`: integer clamped 5..120; null if not a positive number.
- `mainIngredient`: lowercased trimmed string, null if empty.
- On any structural failure (entire `search_hints` missing or wrong shape): drop the whole hints object and fall through to title-only search. Never throw.

### LOCKED — Persistence
- Migration: `ALTER TABLE meals ADD COLUMN search_hints jsonb;`
- `writeMealToDB` (`generate-draft/index.ts`) writes normalized hints.
- Regenerate-meal path (`trpc/index.ts`) writes normalized hints.
- `mealPlan.get` query passes the column through unchanged. Frontend ignores it; consumed only at enrich time.
- Backwards compatible: existing rows have NULL `search_hints` and fall through to title-only behavior.

### LOCKED — Enrichment Consolidation
Replace `findBestSpoonacularMatch` + 2-call sequence with one `searchAndEnrich` function:
- Single `complexSearch` with `addRecipeInformation+addRecipeNutrition` returns full recipe inline (extendedIngredients, nutrition, analyzedInstructions).
- One retry with soft filters dropped if first call returns zero results.
- Returns null if both attempts fail (no further widening).

### LOCKED — Usage Event Cleanup
- Delete the second `insertUsageEvent` call previously tagged `recipes/{id}/information`.
- New consolidated call logs ONE event tagged `recipes/complexSearch?addRecipeInformation` (or analogous tag matching the actual URL).

### LOCKED — Telemetry KPIs (Dev Diagnostics)
Three new counters surfaced on dev page:
1. **First-call hit rate** = `count(cache_hit=false AND retryWithRelaxedFilters=false) / total enrichments`. Target ≥ ~80%.
2. **Enum drop rate** = count of `cuisine`/`type` values from LLM that failed enum check. Log once per drop.
3. **Empty instructions rate** = count of recipes returned with `analyzedInstructions=[]`.

If first-call hit rate < ~80% in production, treat as signal to fix upstream LLM prompt — never relax filters.

### LOCKED — Cache & Quota Constraints
- Cache-first reuse stays mandatory. Existing `spoonacular_cache` schema unchanged.
- Free-tier daily quota: 50 points. Per consolidated call: 1 + 0.01 + 0.025 + 0.025 ≈ 1.06 pt. Theoretical ceiling ~47 enrichments/day from cold cache (vs ~16 today). Cache hit cost: 0.
- Keep `instructionsRequired=true` (free flag, prevents broken instructions mapping per spike 005).
- Keep post-fetch `extractBlockedTerm` defense-in-depth.

### LOCKED — Implementation Order (per PRD "What to Avoid")
**Do NOT roll out 008 (call consolidation) without 006 + 007 first.** The consolidated call costs slightly more per call; the win comes from removing the variant ladder, which only fires reliably once 006 (param mapping) + 007 (search_hints) are in place. Plans MUST sequence: enums/mappers → household aggregator → LLM prompt + persistence → consolidated call.

### Claude's Discretion
- Exact module location: extract to `_shared/spoonacular-mappings.ts` shared between `generate-draft` and `trpc`, OR duplicate inline in each function dir. Recommendation: extract to `_shared/` for single-source-of-truth on the enum tables.
- Migration filename / numbering: follow existing `supabase/migrations/` convention.
- Telemetry storage: where to emit the soft KPIs (existing usage_events table vs dedicated counter table) — choose minimal change.
- Test fixtures: cassette-style HTTP fixtures for `complexSearch` responses or hand-written stubs — pick whatever the existing test harness already uses.
- Plan/wave breakdown: planner picks granularity (likely 4–6 plans grouped into 2–3 waves given the strict dependency chain).
