## User Constraints (from CONTEXT.md)

### Locked Decisions
- **LLM (`grok-4-1-fast-non-reasoning`) is the source of truth for recipe-specific search hints.** Per-recipe cuisine, type, ingredients, ready-time live in the LLM output.
- **Household table is the source of truth for cross-recipe restrictions.** `intolerances`, `excludeIngredients`, `diet`, `equipment` come from aggregated household members.

- `CUISINE_ENUM` ŌĆö 26 values (african through vietnamese ŌĆö full list in PRD).
- `TYPE_ENUM` ŌĆö 14 values (main course, side dish, dessert, ŌĆ” drink).
- `DIET_ENUM` ŌĆö 11 values (gluten free, ketogenic, vegetarian, ŌĆ”, whole30).
- `INTOLERANCE_ENUM` ŌĆö 12 values (dairy, egg, gluten, ŌĆ”, wheat).
- `EQUIPMENT_ENUM` ŌĆö subset of ~12 values matching what UI dropdown actually offers.
- `MEAL_TYPE_TO_SPOONACULAR` map: `breakfastŌåÆbreakfast`, `lunchŌåÆmain course`, `dinnerŌåÆmain course`.

- **HARD filters (never relaxed in retries):** `diet`, `intolerances`, `excludeIngredients`. A "no match" outcome is correct when household constraints can't be met.
- **SOFT filters (dropped on retry):** `equipment`, `maxReadyTime`. One retry path; if hard filters still return zero, return null.
- **Always present:** `query=meal.title`, `number=1`, `instructionsRequired=true`, `sort=popularity`, `addRecipeInformation=true`, `addRecipeNutrition=true`.

- `intolerances`: union across members; allergies matching enum ŌåÆ `intolerances`, non-enum allergies ŌåÆ `excludeIngredients`.
- `excludeIngredients`: union of all member `avoidances` + non-enum allergies (lowercased, trimmed).
- `diet`: single enum value when ALL members agree on same diet; otherwise null. Never pass partial multi-member diet.
- `equipment`: lowercase appliance names that match `EQUIPMENT_ENUM`; non-matches dropped.
- `maxReadyTime`: derived from skill level ŌĆö `beginner=30`, `intermediate=60`, `advanced=null`.

- `cuisine`/`type`: trimmed lowercase, dropped to null if not in enum. Never coerce/fuzzy-match.
- `includeIngredients`: max 4 strings, lowercased, trimmed, empty filtered.
- `excludeIngredients`: max 8 strings, lowercased, trimmed, empty filtered.
- `maxReadyTime`: integer clamped 5..120; null if not a positive number.
- `mainIngredient`: lowercased trimmed string, null if empty.
- On any structural failure (entire `search_hints` missing or wrong shape): drop the whole hints object and fall through to title-only search. Never throw.

- Migration: `ALTER TABLE meals ADD COLUMN search_hints jsonb;`
- `writeMealToDB` (`generate-draft/index.ts`) writes normalized hints.
- Regenerate-meal path (`trpc/index.ts`) writes normalized hints.
- `mealPlan.get` query passes the column through unchanged. Frontend ignores it; consumed only at enrich time.
- Backwards compatible: existing rows have NULL `search_hints` and fall through to title-only behavior.

- Single `complexSearch` with `addRecipeInformation+addRecipeNutrition` returns full recipe inline (extendedIngredients, nutrition, analyzedInstructions).
- One retry with soft filters dropped if first call returns zero results.
- Returns null if both attempts fail (no further widening).

- Delete the second `insertUsageEvent` call previously tagged `recipes/{id}/information`.
- New consolidated call logs ONE event tagged `recipes/complexSearch?addRecipeInformation` (or analogous tag matching the actual URL).

- Cache-first reuse stays mandatory. Existing `spoonacular_cache` schema unchanged.
- Free-tier daily quota: 50 points. Per consolidated call: 1 + 0.01 + 0.025 + 0.025 Ōēł 1.06 pt. Theoretical ceiling ~47 enrichments/day from cold cache (vs ~16 today). Cache hit cost: 0.
- Keep `instructionsRequired=true` (free flag, prevents broken instructions mapping per spike 005).
- Keep post-fetch `extractBlockedTerm` defense-in-depth.

- **Do NOT roll out 008 (call consolidation) without 006 + 007 first.** The consolidated call costs slightly more per call; the win comes from removing the variant ladder, which only fires reliably once 006 (param mapping) + 007 (search_hints) are in place. Plans MUST sequence: enums/mappers ŌåÆ household aggregator ŌåÆ LLM prompt + persistence ŌåÆ consolidated call.

The project already has the right insertion points for this phase: `generate-draft/index.ts` writes streamed meal rows, `trpc/index.ts` owns enrich and regenerate behavior, `mealPlan.get` already hydrates persisted meal rows, and `spoonacular_usage` already captures cache/live usage. [CITED: https://spoonacular.com/food-api/docs#Search-Recipes-Complex] SpoonacularŌĆÖs docs also publish the canonical diet, intolerance, cuisine, and meal-type vocabularies that match the locked server-side enum strategy in the phase context. ### Pattern 1: Shared Normalization Boundary

**What:** Normalize all enum and ingredient hint values exactly once in server code before anything is persisted or sent to Spoonacular. [VERIFIED: codebase grep]
**When to use:** Every draft write, single-slot regenerate, and enrich read path. [VERIFIED: codebase grep]
**Example:**

**What:** Keep `diet`, `intolerances`, and `excludeIngredients` fixed across both attempts, and only drop `equipment` plus `maxReadyTime` on the second attempt. [CITED: https://spoonacular.com/food-api/docs#Search-Recipes-Complex] [VERIFIED: codebase grep]

### Pattern 3: Cache-First Before Network

**What:** Preserve the current two cache checks before any Spoonacular call: existing enriched meal row first, shared `spoonacular_cache` second. [VERIFIED: codebase grep]

### Anti-Patterns to Avoid

- **Do not fuzzy-coerce invalid `cuisine` or `type` values:** The locked behavior is drop-to-null, not best-effort guessing. [VERIFIED: codebase grep]
- **Do not widen hard household restrictions on retry:** That converts a correct `no match` into a safety bug. [VERIFIED: codebase grep]
- **Do not add a second Spoonacular call after the consolidated search:** The quota win disappears immediately. [CITED: https://spoonacular.com/food-api/docs#Search-Recipes-Complex] [VERIFIED: codebase grep]

## Common Pitfalls

### Pitfall 1: Measuring the wrong quota win
**What goes wrong:** A plan counts only the removal of `/recipes/{id}/information` and forgets that current search also widens across a title-query ladder. [VERIFIED: codebase grep]
**Why it happens:** The existing helper `findBestSpoonacularMatch` loops across several title variants before enrichment. [VERIFIED: codebase grep]
**How to avoid:** Replace the ladder with a single, richer `complexSearch` request after mappings and hints are in place. [VERIFIED: codebase grep] [CITED: https://spoonacular.com/food-api/docs#Search-Recipes-Complex]
**Warning signs:** Multiple `recipes/complexSearch?query=` usage rows still appear for one enrich action after implementation. [VERIFIED: codebase grep]

### Pitfall 2: Treating `search_hints` as trusted data
**What goes wrong:** Bad LLM output becomes persistent query behavior or causes throws during regeneration and draft writes. [VERIFIED: codebase grep]
**Why it happens:** Current prompt outputs are raw JSON, and neither write path currently knows about `search_hints`. [VERIFIED: codebase grep]
**How to avoid:** Normalize before insert, clamp numeric fields, drop invalid enums to `null`, and drop the whole object on structural failure. [VERIFIED: codebase grep]
**Warning signs:** Failed writes, non-null invalid `cuisine` strings in the database, or enrich requests using mixed-case free text. [VERIFIED: codebase grep]

### Pitfall 3: Relaxing the wrong filters on retry
**What goes wrong:** A retry returns a recipe that violates allergies or explicit avoidances. [VERIFIED: codebase grep]
**Why it happens:** Equipment and ready-time are convenience filters, but `diet`, `intolerances`, and `excludeIngredients` are safety boundaries. [VERIFIED: codebase grep]
**How to avoid:** Compute hard and soft filters separately in the shared aggregator and reuse that contract inside `searchAndEnrich`. [VERIFIED: codebase grep]
**Warning signs:** Retry logic reconstructs params from scratch instead of deleting only soft keys. [ASSUMED]

### Pitfall 4: Misreading the cache schema during KPI work
**What goes wrong:** A query treats `instructions` as `jsonb` even though the cache stores it as `text[]`. [VERIFIED: codebase grep]
**Why it happens:** Several other recipe fields are `jsonb`, so the array type is easy to assume incorrectly. [VERIFIED: codebase grep]
**How to avoid:** Compute empty-instructions counts with array semantics, not `jsonb_array_length`. [VERIFIED: codebase grep]
**Warning signs:** Dev diagnostics fail against local Supabase or require casts on `instructions`. [VERIFIED: codebase grep]
