## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Create account |
| Empty state heading | Start with your email |
| Empty state body | Create an account or sign in to continue to household setup. If you already began, return with the same email address. |
| Error state | We couldn't complete that request. Check your email and password, then try again. If you're resetting a password, request a fresh link and reopen it from the newest email. |
| Destructive confirmation | Sign out: no confirmation modal; sign out immediately, clear the active session, and redirect to `/auth` with a small inline notice, "You've been signed out." |

Additional copy rules:
- Keep mode labels verb-first and plain: `Sign in`, `Create account`, `Reset password`.
- Use sentence case everywhere except small editorial eyebrow labels already established in the shell.
- Validation copy should be direct and field-specific: `Enter a valid email address.` / `Password must be at least 8 characters.`
- Password reset request success copy should confirm the next step: `Check your email for a reset link.`

## Interaction Contract

- `/auth` remains the single auth surface and swaps between `sign in`, `create account`, and `reset password` modes in-place. Do not navigate to separate top-level auth pages.
- The default visible mode on signed-out entry is `sign in`. The `create account` action must stay equally prominent via an adjacent mode toggle, not a buried text link.
- Signed-in users who hit `/auth` redirect immediately to `/household` without showing the auth form.
- Unauthenticated users who hit `/`, `/household`, or `/plan/:id` redirect immediately to `/auth`. Preserve intended destination in router state so later phases can reuse it, but Phase 2 still lands successful auth on `/household` per locked decision.
- After successful sign up or sign in, show a short in-place submitting/loading state on the button, then route to `/household`.
- Password reset is two-step but visually continuous:
  1. Request mode on `/auth` collects email and confirms that the reset email was sent.
  2. Return-link completion screen stays in-app and uses the same card/shell language, with a dedicated title and new-password form.
- Logout lives in the authenticated shell header area as a quiet secondary text button or pill on the far-right utility cluster. Do not place logout inside the primary route nav items.
- Form validation triggers on submit first, then on blur for corrected fields. Do not show errors before first user interaction.
- Disable primary submit while request is in flight. Preserve field values on recoverable errors.

---

## Visual Hierarchy

- Primary focal point: the auth card title and the active submit CTA form a single first-read unit in the center panel.
- Secondary focal point: the adjacent `Sign in` / `Create account` mode switcher clarifies which action the user is taking before they scan fields.
- Supporting context: any editorial shell copy, utility logout affordance, and background texture remain peripheral and must not compete with the auth card for contrast or size.
- Reading order must stay stable on all breakpoints: title, mode switcher, field stack, helper/error text, then secondary actions such as `Forgot password?` or `Back to sign in`.

---

## Component Inventory

- Auth shell card: centered translucent panel inside existing editorial frame, max width 32rem on desktop, full width on mobile.
- Mode switcher: two primary modes (`Sign in`, `Create account`) as adjacent pills or segmented buttons; reset mode is entered contextually from `Forgot password?`.
- Form stack: email field, password field, submit button, helper/error block, and contextual secondary actions.
- Reset request state: email-only form plus confirmation message region.
- Reset completion state: new password + confirm password, success confirmation, and route back to sign in if needed.
- Protected-route redirect handling: no separate screen animation or splash; redirect immediately.
