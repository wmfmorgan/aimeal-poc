<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser batch action -> tRPC mutation | Untrusted meal selections trigger authenticated enrichment writes and external API usage |
| tRPC router -> Spoonacular + shared cache | Third-party data and quota headers cross into persisted cache and user-visible recipe content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | E | `meal.enrich` | mitigate | Validate ownership through the caller JWT before any meal update and keep Spoonacular API usage entirely server-side |
| T-06-06 | T | shared cache writes | mitigate | Use a separate service-role client only for shared-cache writes and usage-log inserts; keep user-owned meal updates on the caller client |
| T-06-07 | D | batch hook | mitigate | Track per-meal pending/error state and invalidate the plan on each success so one failure cannot stall the rest of the batch |
| T-06-08 | I | `/dev` usage query | mitigate | Return summary and call metadata only; never expose API keys, raw auth headers, or service-role details |
</threat_model>
