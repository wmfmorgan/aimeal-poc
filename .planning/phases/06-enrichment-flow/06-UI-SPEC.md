# Phase 6 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual Tailwind + CSS custom properties) |
| Preset | not applicable |
| Component library | none — bespoke React components built on Tailwind utilities |
| Icon library | none required for Phase 6; prefer text labels and visible check affordances over icon-only controls |
| Font | Newsreader (display/headlines) + Manrope (UI, labels, body) |

Source: Phase 5 UI contract, `tailwind.config.ts`, `src/styles/globals.css`, current generation components, and absence of `components.json`.

Phase 6 must preserve the existing editorial product language:
- Keep the warm paper canvas, glassy white cards, sage primary action color, and spacious rhythm established in Phases 4-5.
- Do not introduce shadcn, Radix primitives, dense dashboard tables, or a separate recipe-management visual system.
- Enrichment should read as an upgrade path layered onto the current grid-management surface, not a new workflow on a different page.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Selection ticks, inline chip gaps, nutrition key/value spacing |
| sm | 8px | Compact status-chip padding, small content gaps, checkbox-label breathing room |
| md | 16px | Default internal spacing, button gaps, flyout list item rhythm |
| lg | 24px | Card padding, section padding, action-bar padding |
| xl | 32px | Major layout gaps between page sections and flyout blocks |
| 2xl | 48px | Full flyout section breaks, hero-to-content separation |
| 3xl | 64px | Top-level page rhythm between header shell and grid/dev sections |

Exceptions:
- Minimum touch target remains `44px` for all clickable surfaces, including `Select meal`, `Select all`, `Enrich selected meals`, `Retry enrichment`, `View recipe`, and flyout close actions.
- Meal cards keep the existing rounded shell at `1.5rem` to `1.75rem`.
- Batch action bar may use a sticky bottom offset and non-scale shadow treatment so selection actions remain visible while scrolling.
- Flyout width may continue to use `min(36rem, 100vw)` for readability on desktop while still filling mobile width.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.75 |
| Label | 14px | 600 | 1.25 |
| Heading | 20px | 600 | 1.2 |
| Display | 30px to 36px | 400 | 1.0 to 1.1 |

Type rules:
- Keep weights to exactly `400` and `600`.
- Card titles remain editorial, using Newsreader at `text-xl` in-grid and `text-3xl` in the flyout.
- Selection counts, status chips, section labels, and nutrition labels use Manrope `14px/600`.
- Ingredients, instructions, and dev-page supporting copy use body text with generous line height rather than compact dashboard typography.
- Numbered instruction steps may use label weight for the step number, but the instruction sentence itself remains body weight.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#faf9f8` with existing warm gradient | App background, flyout backdrop base, page canvas |
| Secondary (30%) | `rgba(255,255,255,0.52)` to `rgba(255,255,255,0.72)` | Grid shell, meal cards, flyout sections, dev cards, sticky action bar base |
| Accent (10%) | `#4A6741` (`--color-sage`) | Reserved for selection-active and enrichment-primary emphasis only |
| Destructive | `#803b26` | Failure states, destructive copy, destructive confirmations only |

Accent reserved for:
- `Enrich selected meals` primary batch CTA
- Selected card outline, checkbox fill, and selection count emphasis
- Positive `Enriched` status chip
- Focus rings and keyboard focus states
- Flyout `View recipe`/recipe-primary affordances if a distinct CTA is needed
- Dev-page usage progress fill when within daily limit

Accent must not be used for:
- Generic body text
- Entire card backgrounds
- All secondary controls
- Failure states

Destructive usage:
- Enrichment failure callout text on affected cards
- Retry-adjacent warning copy only when the issue indicates a failed call
- Existing delete/destructive affordances inherited from Phase 5

Status color contract:
- `Draft`: neutral muted text on pale secondary chip
- `Enriching`: sage-tinted chip with subtle pulse
- `Enriched`: sage solid or high-contrast sage-tinted chip
- `Failed`: destructive-tinted inline surface plus explicit retry action

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Enrich selected meals` |
| Empty state heading | `No enriched recipes yet` |
| Empty state body | `Select one or more draft meals, then enrich them to pull in ingredients, instructions, nutrition, and a recipe image.` |
| Error state | `We couldn't enrich this meal. Retry this card, or keep working on the rest of the plan.` |
| Destructive confirmation | `Clear selection`: `Remove all selected meals from this batch? Your draft meals stay in the plan.` |

Required supporting copy:
- Selection mode entry action: `Select meals`
- Selection mode exit action: `Done selecting`
- Select-all action: `Select all`
- Batch helper text: `{N} meals selected`
- Draft status chip: `Draft`
- Loading status chip: `Enriching`
- Complete status chip: `Enriched`
- Card-level failure CTA: `Retry enrichment`
- Flyout summary label for draft content: `Why this was chosen`
- Flyout recipe section labels: `Ingredients`, `Instructions`, `Nutrition`
- Dev-page section heading: `Spoonacular Usage`
- Dev-page budget summary copy: `Today's usage`

Copy rules:
- Use `recipe` only when real Spoonacular data exists; keep `meal` for draft-only states.
- Do not mention cache internals in grid copy. Cache-hit behavior is invisible in product UX.
- Batch actions should read as deliberate and user-controlled, never automatic or background-only.
- Failure copy must stay local and calm; avoid alarming language like `system failure` or `batch aborted`.

## Component Inventory

### `PlanPage`
- Keep `/plan/:id` as the durable management route.
- Add selection-mode controls above the grid, inside the existing plan-shell section rather than as a new page header.
- Preserve `Create new plan` as secondary to the current-plan workflow.
- Batch-level status belongs in the plan shell or sticky action bar, never as a full-page blocker.

### `MealPlanGrid`
- Preserve the validated Phase 5 layout:
  - Mobile: stacked day sections with day headings
  - Desktop: day columns with meal-type row labels
- Add a grid-wide selection mode without changing slot ordering or collapsing the current layout.
- While selection mode is off, cards behave exactly like Phase 5 management cards.
- While selection mode is on:
  - every filled draft or previously enriched card exposes a clear selection affordance
  - the grid shows a visible `Select all` control
  - the selected count is always visible near the batch CTA
- Empty slots are never selectable and retain their Phase 5 regenerate treatment.

### `MealCard`
- Keep the existing editorial shell and add enrichment metadata rather than replacing the card layout.
- Recommended structure:
  - Eyebrow: meal type
  - Title
  - Short description
  - Status row: `Draft`, `Enriching`, or `Enriched`
  - Actions row
- Required actions in resting state:
  - `View details` for draft cards
  - `View recipe` or `View details` for enriched cards, with flyout content deciding the depth
  - `Regenerate meal`
  - `Delete meal`
- Selection treatment:
  - Add a visible checkbox-like affordance in the card chrome or first action row
  - Selected cards gain an accent outline or ring, not a fully tinted background
  - Selection affordance must remain keyboard-focusable and screen-reader-labeled
- Enriching treatment:
  - keep card shell visible
  - disable destructive/regenerate actions for that card while enrichment is in flight
  - show `Enriching` chip and loading copy without blanking the card
- Enriched treatment:
  - add a positive `Enriched` chip
  - retain title and short description
  - optionally reveal a small recipe-data teaser line such as image/ingredients available, but do not overcrowd the card
- Failure treatment:
  - failed cards show inline error copy plus `Retry enrichment`
  - failure never clears successful cards elsewhere in the batch

### `SelectionActionBar`
- New component or dedicated render path.
- Appears only when selection mode is active.
- Required content:
  - selection count
  - `Select all`
  - `Done selecting`
  - primary CTA `Enrich selected meals`
- Behavior:
  - stays visible as the user scrolls long plans
  - disables the primary CTA when zero meals are selected
  - remains interactive while some selected cards are already enriching, unless the full selected batch has been submitted and no further action is valid

### `MealDetailFlyout`
- Keep the existing right-side slide-over shell and focus management behavior from Phase 5.
- Expand content hierarchy to be recipe-first for enriched meals:
  - hero image
  - ingredients
  - instructions
  - nutrition summary
  - draft rationale lower in the panel
- For non-enriched meals, keep the Phase 5 summary/rationale treatment and add guidance that enrichment unlocks the full recipe view.
- Required enriched sections:
  - Header with meal type, day, title, and status
  - Hero image block
  - Ingredients list using readable ingredient strings
  - Numbered instruction list
  - Nutrition summary grid with 4-6 key nutrients surfaced first
  - `Why this was chosen` section using the original draft rationale
  - Management actions
- Ingredient and instruction sections must remain readable even when Spoonacular returns long content.
- If instructions are empty, show a calm fallback note instead of an empty container.

### `DevPage`
- Replace the Spoonacular placeholder card with a real reporting surface in the same editorial card language as `LLM Requests`.
- Use a two-layer layout:
  - top summary row for daily points used versus 50-point limit, requests made, cache hits, cache misses
  - lower list or table-like card stack for per-call breakdown
- This remains a diagnostics surface, but it must match the rest of the app rather than switching to raw admin styling.

## Interaction States

### Grid-level states

| State | What the user sees |
|-------|--------------------|
| Default management mode | Phase 5 grid behavior with no selection controls visible by default |
| Selection mode, none selected | Visible selection affordances plus disabled `Enrich selected meals` |
| Selection mode, some selected | Selected cards outlined, count visible, batch CTA enabled |
| Batch submitted | Selected cards transition independently into `Enriching`; completed cards flip to `Enriched` as each result lands |
| Partial failure | Successful cards stay enriched; failed cards show local retry affordance |

### Card-level states

| State | Behavior |
|-------|----------|
| Draft | Neutral chip, card selectable, flyout shows draft summary and rationale |
| Selected | Accent outline/ring plus checked affordance; no layout shift |
| Enriching | Loading chip and disabled conflicting actions on that card only |
| Enriched | Positive chip and recipe-available detail path |
| Failed enrichment | Inline destructive-tinted message and `Retry enrichment` button |

### Flyout states

| State | Behavior |
|-------|----------|
| Draft card open | Summary + rationale first, recipe sections absent, enrichment guidance visible |
| Enriched card open | Hero image, ingredients, instructions, nutrition, then rationale |
| Enriching card open | Flyout stays open and reflects in-flight progress without losing context |
| Recipe data missing a section | Section remains present with fallback copy instead of collapsing the information architecture |

### Dev-page states

| State | Behavior |
|-------|----------|
| No usage yet | Summary card shows `0 / 50 points used today` and explains that enrichment calls will appear here |
| Loading | Lightweight inline loading copy within the Spoonacular section |
| Data available | Summary metrics followed by per-call breakdown cards/rows |
| Load failure | Local error message inside the Spoonacular card; LLM logs section remains unaffected |

Accessibility requirements:
- Selection affordances must announce checked and unchecked state programmatically.
- `Select all` must operate on the currently visible plan slots only, matching the user decision in Phase 6 context.
- Focus should remain stable when cards flip from `Draft` to `Enriched`; avoid remount patterns that throw keyboard users to the page root.
- The flyout continues to trap focus and returns it to the originating trigger on close.
- Status changes that occur live after enrichment should be announced with polite live-region messaging where appropriate.

## Layout Guidance

### `/plan/:id` composition

```text
AppFrame
  Plan header card
    eyebrow
    plan title + Create new plan
    plan metadata
  Meal grid card
    selection action bar (conditional)
    meal grid
  Meal detail flyout (conditional)
```

Layout rules:
- Keep the plan header and meal-grid shell as separate cards, as in Phase 5.
- Place the selection action bar inside the meal-grid shell, above the first day row/column.
- Do not move the flyout trigger or batch CTA outside the main plan composition.
- On mobile, the selection action bar may stack into two rows, but `Select all`, count, and primary CTA must remain visible without horizontal scrolling.

### Flyout composition

```text
Flyout panel
  top row: slot context + close
  title
  status
  hero image
  ingredients
  instructions
  nutrition
  rationale
  management actions
```

Layout rules:
- Hero image should use a rounded editorial frame and sit high in the panel.
- Ingredients and instructions are separate blocks, never mixed into the same paragraph.
- Nutrition summary should be compact and scan-friendly, not a long raw dump of every nutrient.
- Draft rationale moves lower than recipe content for enriched meals, but it remains visible in the same panel.

### `/dev` Spoonacular section

```text
Editorial card
  eyebrow
  heading
  short explainer
  summary metric row
  per-call breakdown list
```

Layout rules:
- Keep card styling matched to the existing `LLM Requests` section.
- Use card stacks or structured rows instead of a dense bordered data table.
- The daily budget posture should be immediately visible near the heading without opening a disclosure.
