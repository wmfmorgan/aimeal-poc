# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual Tailwind + CSS custom properties) |
| Preset | not applicable |
| Component library | none — bespoke React components built on Tailwind utilities |
| Icon library | none required; prefer labeled actions and clear text chips over icon-only controls |
| Font | Newsreader (display/headlines) + Manrope (UI, labels, body) |

Source: Phase 5 and Phase 6 UI contracts, `src/routes/plan-page.tsx`, `src/components/generation/MealDetailFlyout.tsx`, and the locked Phase 7 context.

Phase 7 must preserve the established editorial product language:
- Keep the warm paper canvas, soft glass surfaces, sage action color, and calm spacing rhythm already validated in Phases 4-6.
- Do not introduce shadcn, Radix primitives, dense grocery-app checklist styling, or a detached library-management interface.
- Finalization, shopping list, and favorites must feel like an extension of the current `/plan/:id` workspace and right-side panel model.
- The emotional tone should read as "commit this plan and keep the recipes worth repeating," not "complete an admin workflow."

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline badge gaps, tiny metadata separators |
| sm | 8px | Chip padding, compact panel controls, inline helper spacing |
| md | 16px | Default internal spacing, button gaps, list rhythm |
| lg | 24px | Card padding, panel section padding, action rows |
| xl | 32px | Major layout gaps between plan-shell sections |
| 2xl | 48px | Panel section breaks, finalized summary spacing |
| 3xl | 64px | Page-level separation between header shell and content sections |

Exceptions:
- Minimum touch target remains `44px` for `Finalize plan`, `View shopping list`, `Copy shopping list`, `Save to favorites`, and `Open favorites`.
- Card and panel shells retain the existing `1.5rem` to `1.75rem` radius family.
- Right-side panel width may continue to use a usability cap around `min(36rem, 100vw)` on desktop while filling mobile width.
- Shopping-list rows may use slightly tighter vertical rhythm than narrative sections, but never below `12px` gap between grouped items.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.75 |
| Label | 14px | 600 | 1.25 |
| Heading | 20px | 600 | 1.2 |
| Display | 30px to 36px | 400 | 1.0 to 1.1 |

Type rules:
- Keep weights to exactly `400` and `600`.
- Page and panel titles remain editorial and recipe-led, using Newsreader for the plan title, shopping-list heading, and favorites-panel heading.
- Counts, state chips, ingredient quantities, and panel controls use Manrope `14px/600`.
- Shopping-list ingredient lines use body sizing with clear quantity emphasis, not table-heavy condensed text.
- Warning copy for discarding draft meals uses body text with calm hierarchy; do not switch into oversized alert typography.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#faf9f8` with existing warm gradient | App background, panel backdrop base, primary canvas |
| Secondary (30%) | `rgba(255,255,255,0.52)` to `rgba(255,255,255,0.72)` | Header shell, shopping-list cards, favorites panel sections, flyout bodies |
| Accent (10%) | `#4A6741` (`--color-sage`) | Reserved for commit/save actions and positive finalized/favorited emphasis |
| Destructive | `#803b26` | Warning copy, destructive confirmations, irreversible state messaging |

Accent reserved for:
- `Finalize plan`
- `View shopping list`
- `Copy shopping list`
- `Save to favorites` and positive favorited state
- Active panel triggers and keyboard focus rings
- Finalized success badge or summary emphasis

Accent must not be used for:
- General body copy
- Entire grocery-list backgrounds
- All controls indiscriminately
- Warning messaging about discarded draft meals

Destructive usage:
- Finalize warning surface when draft meals will be removed
- Confirmation copy for irreversible finalize action
- Any future remove-from-favorites affordance if introduced during planning

Status color contract:
- `Draft`: neutral muted chip
- `Enriched`: sage-tinted positive chip
- `Finalized`: stronger sage emphasis, but still within the calm editorial palette
- `Unavailable`: quiet neutral or subdued disabled styling for favorite affordances on draft meals

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Finalize plan` |
| Secondary finalized CTA | `View shopping list` |
| Favorites entry CTA | `Open favorites` |
| Favorite action | `Save to favorites` |
| Favorite disabled helper | `Enrich this meal to save it` |
| Favorites empty-state heading | `No saved recipes yet` |
| Favorites empty-state body | `Save any enriched recipe from your plan or flyout to build a favorites library you can revisit across future meal plans.` |
| Shopping list empty-state heading | `No shopping list yet` |
| Shopping list empty-state body | `Finalize this plan to build a de-duplicated shopping list from your enriched meals.` |
| Error state | `We couldn't update this plan. Try again in a moment.` |
| Destructive confirmation | `Finalize plan:` `Draft meals will be removed from this finalized plan. Only enriched meals will be kept.` |

Required supporting copy:
- Finalize helper before confirmation: `Finalize turns your enriched meals into a clean shopping list and locks this plan for cooking.`
- Finalize success summary: `Your shopping list is ready.`
- Shopping-list copy action success: `Shopping list copied`
- Favorites panel heading: `Favorites library`
- Favorites panel helper: `Saved recipes stay available across future meal plans.`
- Favorited state label: `Saved`
- Draft-meal disabled state label: `Favorites are available after enrichment`

Copy rules:
- Use `recipe` when referring to enriched and favoritable meals; use `meal` for draft states.
- Finalization copy must state the discard behavior plainly. Do not hide it behind euphemistic language.
- Shopping-list copy should read like a practical grocery artifact, not a nutrition report or internal JSON summary.
- Favorites copy should emphasize reuse and continuity, not collection or curation for its own sake.

## Component Inventory

### `PlanPage`
- Keep `/plan/:id` as the durable plan-management route.
- Extend the existing header shell with plan-level finalization controls rather than creating a separate checkout-style screen.
- Required plan-level actions:
  - `Finalize plan` while the plan is not finalized
  - `View shopping list` once a shopping list exists
  - `Open favorites` as a panel/flyout trigger, not a top-level route change
- If the plan is already finalized, replace the primary finalize CTA with a finalized-status summary plus `View shopping list`.

### `PlanFinalizationCard`
- New plan-level summary block inside the existing page shell.
- This is the primary visual anchor on `/plan/:id` until the plan is finalized.
- Required content:
  - concise explanation of what finalization does
  - count summary for enriched meals versus draft meals if available
  - clear warning when draft meals will be discarded
  - primary CTA `Finalize plan`
- This should feel like a considered commitment surface, not a red alert box.

### `FinalizePlanConfirmation`
- Modal or inline panel treatment is acceptable during planning, but the visual language must stay consistent with the editorial shell.
- Required content:
  - confirmation title
  - plain-language discard warning
  - confirmation CTA
  - cancel action
- If implemented as a modal, it should feel lighter than a system dialog and must not use harsh dashboard chrome.

### `ShoppingListPanel`
- Right-side panel or flyout launched from the plan screen.
- Must remain connected to the existing grid workspace so the plan is still the anchor context.
- Required content:
  - list heading and copy action
  - grouped ingredient sections using Spoonacular categories when available
  - graceful fallback grouping when categories are missing
  - one ingredient line per de-duplicated item with rolled-up quantity totals
- Avoid spreadsheet-like dense rows; each item should remain easy to scan on mobile.

### `ShoppingListGroup`
- Section render for ingredient categories such as produce, dairy, pantry, or fallback groups.
- Group headings should be visible but quiet.
- Quantity formatting should emphasize combined totals first, with any secondary detail kept subtle.

### `MealCard`
- Preserve the Phase 5/6 shell and slot behavior.
- Add favorite affordances only for enriched meals.
- Draft cards must not expose an active favorite control; use a disabled or absent state with helper text if needed.
- Finalized plans may still show the meals in-grid, but the grid must visually communicate that the shopping-list view is now the practical next step.

### `MealDetailFlyout`
- Keep the existing right-side flyout and recipe-first hierarchy from Phase 6.
- Add `Save to favorites` within the flyout for enriched meals.
- If the current meal is already saved, show a positive persisted state such as `Saved`.
- Include an entry point from the flyout to `Open favorites` only if it does not compete with the main recipe content.

### `FavoritesPanel`
- Panel- or flyout-based library surface launched from the plan page or flyout.
- Must not replace the plan route or become a full-screen primary page for Phase 7.
- Required content:
  - heading and helper copy
  - list of saved recipes with image, title, and lightweight metadata
  - clear empty state when no favorites exist yet
- The panel should feel browsable and calm, not like an asset manager.

## Interaction States

### Plan-level states

| State | What the user sees |
|-------|--------------------|
| Draft plan, mixed meal states | `Finalize plan` visible with helper text explaining that draft meals will be removed if finalized |
| Draft plan, no enriched meals | Finalize surface remains visible but may be disabled with copy directing the user to enrich meals first if implementation requires a minimum viable finalized set |
| Finalize confirmation open | Clear warning, explicit confirm/cancel, no ambiguity about draft-meal removal |
| Finalized success | Calm success summary plus `View shopping list`; no celebratory dashboard treatment |
| Finalized revisit | Existing plan page loads with finalized state visible and shopping-list access available immediately |

### Shopping-list states

| State | Behavior |
|-------|----------|
| Generated and viewable | Right-side panel shows grouped, de-duplicated items with copy action visible |
| Copy action pressed | Panel stays open and surfaces lightweight `Shopping list copied` feedback |
| Missing category metadata | Items still render under graceful fallback headings without blocking the panel |
| No shopping list yet | Empty-state copy explains finalization is required |

### Favorites states

| State | Behavior |
|-------|----------|
| Enriched meal, unsaved | Active `Save to favorites` action visible |
| Enriched meal, saved | Positive saved state shown; action should not appear ambiguous or re-clickable without purpose |
| Draft meal | Favorite affordance absent or disabled with helper text; never pretend the save completed locally |
| Empty library | Panel shows calm empty state pointing the user back to enriched meals |
| Persisted library revisit | Saved recipes appear across plans without needing the original plan context to remain open |

Accessibility requirements:
- Any finalize confirmation treatment must trap focus while open and return focus to the triggering control on close.
- `Copy shopping list` and `Save to favorites` must expose success state to screen readers.
- Disabled favorite affordances on draft meals must still explain why the action is unavailable.
- Shopping-list grouping must remain readable to assistive tech; do not rely on visual separators alone.

## Layout Guidance

### `/plan/:id` page composition

```text
AppFrame
  Plan header shell
    Plan title / metadata
    Create new plan
    Open favorites
  Finalization summary card
    Helper copy
    Finalize plan or finalized status
    View shopping list
  Meal grid shell
    Existing grid and flyout interactions
```

Rules:
- Keep plan-level actions near the top of the page; do not bury finalization below the grid fold.
- `Open favorites` should feel adjacent to plan management, not like a secondary app section.
- Shopping-list and favorites experiences should open from the current context rather than navigate away from it.

### Mobile behavior
- Keep the current stacked day-section rhythm from prior phases.
- Plan-level actions may stack vertically, but preserve clear priority:
  - `Finalize plan` or finalized status first
  - `View shopping list` second
  - `Open favorites` third
- Shopping-list rows must wrap gracefully; do not force narrow table columns.

### Desktop behavior
- Retain the spacious editorial header and grid shell from Phases 5-6.
- Right-side panels for meal details, shopping list, and favorites should share a common shell language.
- Do not open multiple competing panels at once; one focused side panel at a time is the contract.

---

## Non-Goals And Guardrails

- Do not add a top-level `/favorites` route as the primary library surface.
- Do not allow draft meals to be saved as favorites.
- Do not hide the fact that draft meals are discarded on finalize.
- Do not turn the shopping list into an admin table, export console, or nutrition worksheet.
- Do not break the existing plan-grid and flyout interaction model established in Phases 5 and 6.
