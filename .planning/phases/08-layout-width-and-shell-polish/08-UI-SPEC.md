> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual Tailwind + CSS custom properties) |
| Preset | not applicable |
| Component library | none — bespoke React components built on Tailwind utilities |
| Icon library | none required in this phase |
| Font | Newsreader (display/headlines) + Manrope (UI, labels, body) |

Source: `tailwind.config.ts`, `src/styles/globals.css`, `src/app/layout/AppFrame.tsx`, and Phase 7 UI contract.

Phase 8 must preserve the established editorial shell:
- Keep the warm paper background, translucent white shells, soft radius family, and sage action color.
- Make the app feel less margin-heavy, not more utilitarian.
- Do not introduce shadcn, dashboard chrome, data-table framing, or new card/flyout visual language.
- This phase changes shell width, page margins, section spacing, and responsive composition only.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Tiny chip gaps, inline separators |
| sm | 8px | Compact label-to-control spacing, tight grid metadata |
| md | 16px | Default internal spacing, mobile shell gutters, stacked section rhythm |
| lg | 24px | Tablet shell gutters, card padding, default plan-page section gaps |
| xl | 32px | Desktop shell gutters, major route-level gaps |
| 2xl | 48px | Wide-screen route separation where reading width needs more air |
| 3xl | 64px | Reserved for hero-style display spacing only; not for regular plan-page stacking |

Exceptions:
- Minimum touch target remains `44px` for nav pills, shell actions, and existing plan actions.
- Rounded shell family remains `1.5rem` to `2rem`.
- `/plan/:id` may tighten internal section rhythm to `24px` on desktop and `16px` on mobile; do not keep the current overly loose `32px` stack everywhere.
- Desktop grid gutters may tighten by day count: `24px` for 4 days, `16px` for 5 days, `8px` for 6-7 days.

- Non-plan routes may keep wider reading line-lengths, but body copy should not exceed calm article-like measure.

## Layout Contract

### App shell

- `AppFrame` is the primary Phase 8 shell refactor point.
- Reduce outer viewport padding to `16px` mobile, `24px` tablet, `32px` desktop.
- Increase shell usable width from the current `max-w-6xl` to an editorial wide-shell target of `max-w-[88rem]`.
- Keep the shell centered and rounded; do not move to full-bleed.
- Reduce shell inner padding to `16px` mobile, `24px` tablet, `28px` desktop so reclaimed width benefits content, not only chrome.
- Header-to-main spacing should tighten from the current large gap to `24px` mobile and `32px` desktop.

### Shared route composition

- Apply the slimmer shell and tighter vertical rhythm to `/`, `/household`, `/plan/:id`, `/plan/new`, `/dev`, and `/auth`.
- Reading-heavy routes should keep content measure capped inside the wider shell:
  - `/auth`: form shell max `32rem`
  - `/household`: primary form rail max `52rem`, supporting rail max `24rem`
  - `/`: hero copy max `42rem`
  - `/dev`: content fills shell width, but each data block keeps its own card framing
- Shared page stacks should default to `24px` vertical separation on mobile and `32px` on desktop, not blanket `space-y-8` everywhere.

### Plan page hierarchy

- `/plan/:id` is the primary beneficiary of reclaimed width.
- The plan header, finalization card, and selection bar must remain visually subordinate to the meal grid.
- Header actions stay compact and right-aligned on desktop; they should not create a second dominant column.
- `PlanFinalizationCard` may rebalance into a more efficient two-column layout on desktop, but its text block should cap around `40rem`.
- The grid wrapper should use the full available content width inside the page shell.

## Responsive Composition

### Desktop `>= 1280px`

- Show the desktop matrix grid as the default plan view.
- Target full 7-day readability without horizontal overflow in the widened shell.
- Use adaptive grid density:
  - 4 days: relaxed gutters and section padding
  - 5 days: moderate tightening
  - 6-7 days: visibly tighter gutters and labels, while cards remain readable
- Day labels stay centered above columns; meal-type labels stay as the left rail.

### Tablet `768px-1279px`

- Tablet must improve meaningfully; do not treat it as a slightly larger mobile stack by default.
- Keep the matrix layout for the plan grid on tablet when it remains readable after density tightening.
- Tighten plan-page section padding before allowing overflow.
- Horizontal overflow is an allowed fallback only after tablet grid gutters hit their tighter contract and the grid would otherwise collapse below readable card width.
- Non-plan routes should reduce decorative whitespace first and preserve clear single-column reading flow unless a second column materially helps.

### Mobile `< 768px`

- Keep the existing stacked plan structure.
- Reduce shell padding and inter-section gaps so mobile feels tighter, not redesigned.
- Day sections remain grouped vertically with meal-type labels above cards.
- Overlays and flyouts keep existing full-width mobile behavior.

## Component Scope Guardrails

- `MealPlanGrid`: adjust container width, grid column math, label spacing, and breakpoint behavior only.
- `PlanFinalizationCard`: adjust proportion, alignment, and padding only; do not redesign its actions or copy hierarchy.
- `FavoritesPanel` and `MealDetailFlyout`: preserve current width model and capabilities; only ensure shell changes do not make them feel disconnected from the widened workspace.
- `MealCard` content, controls, and detail density are explicitly deferred to Phase 9.
- No backend, generation, enrichment, or favorites logic changes belong in this phase.

---

## Route-by-Route Application

- `/`: keep the editorial split layout, but let the main narrative column breathe by using the wider shell and slightly tighter card spacing.
- `/household`: reduce oversized horizontal margins around the main form and tighten section stacks so long forms feel more continuous.
- `/plan/new`: reuse the wider shell and tighter section rhythm from the persisted plan page so generation mode does not feel like an older layout system.
- `/plan/:id`: prioritize grid width first, then tighten surrounding sections to support it.
- `/dev`: use the wider shell so logs and usage cards stop feeling boxed in, while preserving card boundaries around technical content.
- `/auth`: keep the focused single-surface composition; shell changes should mainly reduce wasted perimeter space rather than widen the form excessively.
