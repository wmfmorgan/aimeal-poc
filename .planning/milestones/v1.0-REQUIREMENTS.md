# Requirements Archive: PlanPlate v1.0

**Archived:** 2026-04-23  
**Milestone Outcome:** ✅ Shipped with acknowledged closeout gaps  
**Core Value:** Users see a complete draft meal plan in under 2 seconds (via streaming), then stay in control of exactly which meals get enriched with full recipes.

## Outcome Legend

- `Complete` — shipped and validated during v1.0
- `Adjusted` — original requirement changed before close; shipped outcome differs from original wording

## v1 Requirement Outcomes

### Authentication

- [x] **AUTH-01**: User can sign up with email and password
- [x] **AUTH-02**: User can log in and stay logged in across browser sessions
- [x] **AUTH-03**: User can log out from any page
- [x] **AUTH-04**: User can reset password via email link

### Household Setup

- [x] **HSHD-01**: User can create a household with a name and cooking skill level
- [x] **HSHD-02**: User can add household members with names
- [x] **HSHD-03**: User can set dietary preferences, allergies, and avoidances per member
- [x] **HSHD-04**: User can specify household appliances (Instant Pot, air fryer, etc.)
- [x] **HSHD-05**: User can edit household and member details after creation

### Draft Generation

- [x] **GEN-01**: User can trigger a meal plan draft, selecting how many days and which meal types to include
- [x] **GEN-02**: Draft plan streams to the client — meals appear progressively
- [x] **GEN-03**: Generated meals respect all household allergies and avoidances
- [x] **GEN-04**: Generated meals match household cooking skill level and available appliances
- [x] **GEN-05**: User can regenerate a single meal without regenerating the full plan
- [x] **GEN-06**: Each meal includes title, short description, and LLM rationale

### Meal Plan Management

- [x] **PLAN-01**: User sees a 7×3 grid of meals organized by day and meal type
- [x] **PLAN-02**: User can edit a meal's title inline  
  Outcome: Adjusted. The shipped v1.0 contract intentionally removed inline title editing and locked a no-inline-edit regression guard instead.
- [x] **PLAN-03**: User can delete a meal from the plan
- [x] **PLAN-04**: User can view meal details (description, rationale) from the grid

### Enrichment

- [x] **ENRCH-01**: User can select one or more draft meals to enrich with real recipe data
- [x] **ENRCH-02**: Enrichment fetches ingredients, nutrition, instructions, and image from Spoonacular
- [x] **ENRCH-03**: Previously fetched recipes are served from cache
- [x] **ENRCH-04**: User sees enriched meal data update live in the grid after enrichment completes
- [x] **ENRCH-05**: User can view full recipe details in a flyout panel

### Finalization

- [x] **FINAL-01**: User can finalize a meal plan after reviewing enriched meals
- [x] **FINAL-02**: Finalized plan generates a de-duplicated consolidated shopping list
- [x] **FINAL-03**: User can view and copy the shopping list

### Favorites

- [x] **FAV-01**: User can mark any meal as a favorite
- [x] **FAV-02**: Favorited meals are saved to the user's favorites library across plans

### Dev Tools

- [x] **DEVT-01**: App persists the last 10 LLM prompt + response pairs to the DB
- [x] **DEVT-02**: App tracks Spoonacular API usage per calendar day
- [x] **DEVT-03**: Developer page shows LLM request log
- [x] **DEVT-04**: Developer page shows Spoonacular daily usage details

### Local Dev Environment

- [x] **DEPL-01**: App runs locally via `netlify dev` proxying to local Supabase
- [x] **DEPL-02**: All API keys are loaded from `supabase/functions/.env` for local dev
